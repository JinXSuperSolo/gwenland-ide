# Integrated Terminal (M3): PTY backend, XTerm.js front-end, multi-session, scrollback cap, and error detection

- **Date:** 2026-06-20
- **Issue:** GWEN-244 through GWEN-251
- **Milestone:** Milestone 3 — Integrated Terminal (Waves 1–6)

## Problem / Context

Since M2.5 the Terminal panel had been a grey placeholder sitting below the Workspace — a box with the words "Terminal placeholder — M3" and nothing behind it. M3 was about making it real: a working interactive shell, with a pseudo-terminal backend in Rust streaming to an XTerm.js front-end rendered on a Canvas 2D surface.

The catch is that none of this had been attempted in the project before, and the hard part is cross-platform PTY handling. Windows does not have Unix pseudo-terminals; it has ConPTY, which behaves differently enough that "it works on my Linux box" means very little. So the plan deliberately front-loaded a research step and split the work into six waves, each ending at a point where the app still launches cleanly even if the next wave never gets built. The locked-in decisions from earlier planning carried through: Canvas 2D rendering (not WebGL, not the slow DOM renderer), a ring buffer for scrollback, and reactive per-chunk error detection rather than periodic full re-scans.

The repo layout is the same as M2.5: `frontend/` is itself the Tauri crate (`frontend/src/main.rs`), the shared logic lives in `engine/` which is kept completely free of any Tauri dependency, and the Svelte app lives in `frontend/ui/`. pnpm throughout.

## Change

### Research first (before any code)

The candidate crate was `portable-pty`, the PTY layer extracted from wezterm. Before adding it we checked the things that actually matter: it is MIT-licensed, it picks the right backend at runtime via `native_pty_system()` (ConPTY on Windows, openpty on Unix) so a single binary adapts to wherever it runs, and the read/write/resize/kill surface is genuinely the same shape on both platforms. The dependency footprint is larger than the rest of the lean `engine/` crate (it pulls in `nix`, `libc`, `winapi`, `winreg`, and friends), but most of that is platform-gated and the trade is reasonable for what it buys. The one thing the research flagged up front — and which turned out to be exactly right — was that the Windows and Unix behavior is symmetric in the API but not in the lifecycle, and that there would be ConPTY-specific traps. There were three of them, and they cost the most time in the whole milestone.

### Wave 1 — PTY session manager (engine, zero Tauri)

`engine/src/terminal.rs` is the core: a `PtySession` that spawns the platform default shell (honouring `%ComSpec%` on Windows and `$SHELL` on Unix, falling back sensibly), with create / write / resize / status / kill. It owns the PTY master, the child handle, a kill handle, and a background reader thread that drains output into a shared buffer. No Tauri anywhere — it is a plain byte pipe.

This is where the ConPTY traps surfaced, and the first attempts genuinely hung. The fixes, all of which ended up baked into the code and the tests rather than just noted:

- **The reader never reaches end-of-file on Windows.** On Unix, once the child exits and the master is dropped, a blocking read returns EOF and the reader thread ends. On Windows a blocking read on a ConPTY pipe simply hangs forever once the child is gone — dropping the master does not wake it. The first implementation tried to join that reader thread during teardown, which deadlocked every time. The fix is to deliberately *not* join the reader: the session kills and reaps the child, drops the writer and master, and detaches the reader thread. The thread holds only a cloned read handle and no child handle, so leaving its blocked read to be cleaned up at process exit leaks nothing meaningful, and the captured output is already safe in the buffer.

- **Killing an already-dead child errors on Windows.** A fast command (an `echo` that finishes instantly) can have exited before `try_wait` notices, and calling the kill handle on it returns "The handle is invalid (os error 6)". The teardown now treats a kill failure as success whenever a follow-up `wait` confirms the child is actually gone, and the whole thing is idempotent so an explicit kill followed by a drop never double-faults or hangs.

- **The shell stalls at startup waiting for the terminal to answer it.** This was the most baffling one. A freshly spawned cmd or PowerShell emits an escape sequence asking the terminal for the cursor position (the DSR query, `ESC[6n`) and then *blocks until it gets an answer back*. A headless test harness with no terminal emulator sees only those few bytes and no prompt, no echo, nothing — the shell is simply waiting. A real terminal (XTerm.js, which arrives in Wave 3) answers automatically. The engine deliberately stays a dumb byte pipe and does not interpret escape sequences, so the Wave 1 tests answer the query themselves (writing back a synthetic cursor-position report) to stand in for the renderer, after which the shell proceeds normally and the command output appears.

To stop any future regression of the deadlock from silently hanging the whole test run for minutes, the tests run their bodies under a watchdog that fails fast with a clear label if a body exceeds its deadline. There are tests for spawn-command-output-kill, idempotent kill of a running session, clean teardown when a live session is dropped without an explicit kill, and multi-command input.

One unrelated thing surfaced here that was blocking everything: the workspace root `Cargo.toml` declared a root `[package]` with no `src/lib.rs`, `src/main.rs`, or any target, which made cargo refuse to parse the whole workspace — nothing built, not the engine, not the frontend. It had been getting by on cache. The root was turned into a proper virtual workspace (the `[package]` section removed; `frontend` and `engine` are the real crates), which is the conventional layout and leaves `cargo tauri build` unaffected.

### Wave 2 — I/O bridge to the front-end

The engine reader thread gained an optional per-chunk callback (`OutputCallback`), so output can be pushed live without anyone polling, while still being retained in the buffer. The Tauri side in `frontend/src/main.rs` is a `TerminalManager` holding the live sessions in a map keyed by string id, registered as managed state. The commands are `terminal_create` (returns an id), `terminal_write`, `terminal_resize`, and `terminal_kill`. Output is pushed to the webview as a `terminal://output` event carrying the raw bytes; the renderer decodes them. Typed wrappers and an `onTerminalOutput` listener went into `frontend/ui/src/lib/tauri/commands.ts`. Session ids were introduced here rather than later, so the multi-session work in Wave 4 needed no rework. This is also where it turned out the frontend crate had no direct `serde` dependency (the old commands all returned types whose derives lived in the engine), so `serde` was added to `frontend/Cargo.toml` for the new event payload.

### Wave 3 — Terminal component (XTerm.js + Canvas 2D)

`frontend/ui/src/lib/terminal/xterm-setup.ts` builds the terminal, loads the Canvas addon and the fit addon, and applies a theme matched to the IDE's dark palette (XTerm paints to a canvas and cannot read the CSS `oklch` tokens, so the colours are concrete values approximating them). `Terminal.svelte` is the thin component: it creates the terminal, calls `terminal_create`, pipes PTY output into `term.write`, pipes keystrokes from `term.onData` back to `terminal_write`, and uses a `ResizeObserver` to re-fit and tell the PTY the new dimensions whenever the panel is dragged. This is the structure that mirrors how the editor splits `codemirror-setup.ts` from `Editor.svelte`.

The notable snag here was a version trap. `npm`/`pnpm` happily installed `@xterm/xterm@6.0.0` as the latest, but the Canvas addon's peer dependency is `^5.0.0` and there is no stable 6.x release of it — so the renderer the whole milestone depends on would have been on an unmet peer. XTerm core was pinned to `5.5.0`, which the stable `@xterm/addon-canvas@0.7.0` and `@xterm/addon-fit@0.11.0` are built against, and the unmet-peer warning went away. The DSR startup stall from Wave 1 is a non-issue here because XTerm answers the query itself, so the cmd prompt appears normally.

Verification: the app was launched with `cargo tauri dev`. The process tree confirmed the full chain (`GwenLand-IDE` had spawned `conhost.exe` and a real `cmd.exe` child), and a screenshot showed the panel rendering a live `C:\Users\reyha>` prompt with a blinking cursor instead of the old grey box. The window was resized to exercise the re-fit path without crashing. What could not be done from here is a human typing into the terminal and watching output, and confirming the text reflow after dragging the divider — that still wants a manual eyeball.

### Working-directory fix (found in use)

The terminal opened in the home directory regardless of which folder was open in the IDE — it should start in the project folder, like every other editor's terminal. The fix runs the full length of the stack: `PtySession::spawn` and `spawn_with_callback` take an optional `cwd`, which is applied to the command only if the directory actually exists (a missing path is ignored so the shell still spawns); `terminal_create` gained a `cwd` argument; and `TerminalInstance` reads the open folder from the workspace store at spawn time and passes it through. There is an engine test that points a shell at a fresh temp directory, asks it to print its working directory, and checks the directory name comes back.

### Wave 4 — Multiple sessions and per-session kill

`frontend/ui/src/lib/stores/terminal-sessions.ts` owns the list of sessions and the active one. `Terminal.svelte` was split into `TerminalInstance.svelte` (one shell each) and `TerminalPanel.svelte` (the tab strip, the new-terminal button, the collapse button, and per-tab close-which-kills). Every session stays mounted; switching tabs only toggles visibility, so each session keeps its own scrollback and running process — genuinely isolated state rather than a single view being rebuilt. An instance that becomes visible re-fits and refocuses, because a hidden element cannot be measured. The numbering ("Terminal 1", "Terminal 2") never gets reused when a tab in the middle is closed.

The scope here was deliberately new-and-kill. Clear and restart were left as clean extension points but not built this wave. There is also a known behaviour to be aware of: collapsing the whole panel unmounts everything and kills the sessions; the tab list is remembered and reopening respawns fresh shells, so there is no stale-handle bug, but scrollback is lost on collapse. Persisting across collapse is future work.

This wave is mostly UI, and the verification reflects that: the type-check and build pass, and on launch exactly one shell spawns for the initial session with no leaked processes afterwards. Driving the tab strip with synthetic clicks did not work reliably — Windows refused to let the screenshot/automation steal foreground focus from the browser, and the background dev process was flaky — so opening several tabs, switching between them, and closing one are the things that still want a manual check.

### Wave 5 — Scrollback line cap (ring buffer)

A plain growing `Vec<u8>` would let a `cargo build`, an `ls -R`, or a runaway print loop grow memory without limit. `engine/src/ring_buffer.rs` caps retention by line count (the unit the spec asks for; default 10,000 lines). The interesting part is that PTY output arrives as arbitrary byte chunks that do not line up with line boundaries — a chunk can hold half a line, several lines, or split a line down the middle. The buffer tracks one pending (not-yet-newline-terminated) line and promotes it to a completed line on each newline; once there are more completed lines than the cap, the oldest are dropped first. Bytes are stored verbatim, escape sequences and all, so the rendered scrollback is faithful. `PtySession` now holds this instead of the raw vector. There are unit tests for chunk-boundary reassembly, trimming, the pending line surviving a trim, and a flood test pushing 100,000 lines into a 100-line cap and confirming it stays bounded, plus a session-level stress test that runs a 500-line print loop against a 50-line cap and checks the newest line survives while the oldest is gone.

### Wave 6 — Error detection (partial, by design)

`engine/src/error_detect.rs` scans output reactively, one chunk at a time, for common error signatures, and never re-scans the whole buffer — it keeps the same one-line carry-over trick as the ring buffer and only inspects newly completed lines. It is dependency-free (no regex crate): the signatures are fixed substrings matched against an ANSI-stripped copy of each line, so a colourised `error:` still matches. The pattern set is curated to keep false positives low — generic compiler diagnostics, Rust panics and error codes, Python tracebacks, npm errors, "command not found" and its Windows wording, "no such file" — each tagged with a label. When a line matches, the session records a signal and fires an optional error callback.

The Tauri side forwards each detection as a `terminal://error` event (id, label, and the offending line), and `commands.ts` has an `onTerminalError` listener ready for it. Nothing in the UI consumes it yet — that is intentional. The original issue depends on an AI chat sidebar that does not exist as a milestone, so this wave only detects and exposes the signal; wiring it into a sidebar is for whenever that milestone lands. The acceptance test runs a genuinely failing command (an unknown command on Windows, listing a missing path on Unix) and confirms the flag flips, with a companion test confirming a clean `echo` does not trip it.

## Why this approach

Keeping the PTY logic in `engine/` and only the command-and-event plumbing in `frontend/src/main.rs` preserves the project's existing split: the core is testable in isolation and carries no Tauri dependency, while the bridge stays a thin layer. The per-chunk callback rather than polling is what makes the output streaming non-blocking and keeps the engine ignorant of Tauri. Pushing the ConPTY quirks (reader detachment, kill tolerance, the DSR answer) into code and tests rather than leaving them as lore means the next person does not rediscover them the hard way. The version pin on XTerm is the unglamorous but correct call — the locked decision was Canvas 2D, and the stable Canvas addon only supports the 5.x line. Error detection avoiding a regex dependency and avoiding full re-scans keeps it cheap enough to run on every chunk of a busy build.

## Impact

- The Terminal panel is now a working interactive shell instead of a placeholder, with multiple tabbed sessions, per-session kill, a 10,000-line scrollback cap, and reactive error flagging exposed to the front-end.
- New: `engine/src/terminal.rs`, `engine/src/ring_buffer.rs`, `engine/src/error_detect.rs`; four new Tauri commands plus `terminal://output` and `terminal://error` events in `frontend/src/main.rs`; `frontend/ui/src/lib/terminal/xterm-setup.ts`, `TerminalInstance.svelte`, `TerminalPanel.svelte`, `stores/terminal-sessions.ts`, and the terminal wrappers in `commands.ts`. Removed: the old `TerminalPlaceholder.svelte` and the single-session `Terminal.svelte`.
- The engine test suite stands at 46 passing (PTY lifecycle, ring buffer, and error detection), the UI type-check is clean, and the frontend crate builds. No leaked shell processes after any test run — the kill-and-reap teardown holds end to end.
- The root `Cargo.toml` was repaired into a virtual workspace (it had been quietly broken and is what unblocked all builds). `serde` was added to `frontend/Cargo.toml`. XTerm is pinned to 5.5.0 for Canvas-addon compatibility.
- **Things to keep in mind:** the front-end pieces (Waves 3 and 4) are verified by build, type-check, process-tree, and screenshot, but not yet by a human typing into a terminal, switching tabs, and dragging the divider — that manual smoke test is still worth doing. Per-session Clear and Restart were deferred. Collapsing the terminal panel kills its sessions and loses scrollback (the tab list survives and shells respawn). The `terminal://error` signal has no consumer until an AI chat sidebar exists. Exit-code detection is handled through text patterns rather than reading each sub-command's exit status, since an interactive shell does not surface those without prompt parsing. The error pattern list is curated and lives at the top of `error_detect.rs` for easy tuning. The frontend Rust package is named `GwenLand-IDE` (with a hyphen), so it builds with `cargo build -p GwenLand-IDE`.
