# LSP Bridge (M6): a single generic language-server client for Rust, TS/JS, and Python — diagnostics + autocomplete, BYO servers

- **Date:** 2026-06-22
- **Issue:** GWEN-260 onward (M6 — LSP Bridge)
- **Milestone:** Milestone 6 — LSP Bridge (foundation: diagnostics + autocomplete)

## Problem / Context

M6 adds the first language-intelligence layer: a Language Server Protocol bridge spanning Rust, TypeScript/JavaScript, and Python. The milestone is deliberately broad-but-shallow — one protocol implementation proven against three different servers, but only **diagnostics** (squiggles + gutter markers) and **autocomplete**. Hover, go-to-definition, references, rename, code actions, formatting, semantic tokens, inlay hints, and signature help are explicitly M6.5 and were not built (not even as stubs).

Two scope decisions were locked by the spec and held throughout. First, **BYO servers only**: the IDE detects and connects to executables the user already installed (`rust-analyzer`, `typescript-language-server`, `pyright-langserver`, or a configured `pylsp`) — nothing is bundled, embedded, downloaded, or auto-installed, because a packaged toolchain would blow the binary budget that M1–M5 protected. Second, the **engine/Tauri/UI split** from earlier milestones is preserved exactly: all protocol logic lives in `engine/` with **zero `tauri::` imports**, `frontend/src/main.rs` is the thin command-and-event bridge, and the Svelte app consumes typed wrappers and stores.

The work was taken as six dependency-ordered waves, each ending with the app compiling and earlier behaviour non-regressed: (1) research + foundations, (2) process lifecycle + status, (3) document sync + diagnostics, (4) CodeMirror diagnostics + status UI, (5) completion + autocomplete, (6) settings UI + gated real-server smoke + polish.

## Change

### Wave 1 — Research, types, settings, root detection, JSON-RPC foundation

The first wave locked the implementation choices and built the dependency-free core, documented at the top of `engine/src/lsp/mod.rs`:

- **No `lsp-types`, no `tower-lsp`.** `tower-lsp` is server-oriented and would drag in a large async tower stack for no client benefit; `lsp-types` is usable but large and churny, and M6 touches only a handful of messages. Instead the protocol is hand-rolled in `engine/src/lsp/json_rpc.rs` (Content-Length framing + JSON-RPC envelopes over the `serde_json` we already had), with our own stable UI-facing DTOs. The upshot: **zero new crates were added to `engine/Cargo.toml`** — the stdio process uses `std::process`, and `PATH`/`PATHEXT` resolution is implemented by hand.
- **Full-document sync.** The client advertises `TextDocumentSyncKind::Full` and sends the whole document on every change; the UI debounces. This is far simpler to get right against three servers than incremental sync, and the upgrade path (swap the advertised kind + send range deltas in `did_change`) is left clear.

The module skeleton went in under `engine/src/lsp/`: `language.rs` (`LanguageId` + extension mapping — `.rs`/`.ts`/`.tsx`/`.js`/`.jsx`/`.mjs`/`.cjs`/`.py`, with TS and JS mapping to distinct language ids but a shared `typescript` server bucket via `server_key()`); `config.rs` (`LspSettings`/`LanguageServerSettings`, default commands, and `effective()` resolution where a blank command falls back to the built-in default); `root.rs` (nearest-marker workspace-root detection per language, plus hand-rolled `file://` URI encode/decode covering Windows drive paths and POSIX paths); `json_rpc.rs` (frame encode/decode tolerant of partial and multiple frames in one read, message classification into response/notification/server-request, a monotonic `u64` id generator, and a generic pending-request map with a `drain()` crash-cleanup API); `error.rs` (a serde-tagged, user-safe `LspError` with a `sanitize()` that strips control chars from any server-originated text); and DTO files `diagnostics.rs`/`completion.rs`. `LspSettings` was added to `Settings` with `#[serde(default)]` so M1–M5 settings files without an `[lsp]` table still load.

### Wave 2 — BYO detection, process lifecycle, initialize, status, restart

`engine/src/lsp/process.rs` resolves a configured command (absolute path checked directly; bare names searched on `PATH`, and on Windows extended with `PATHEXT` so npm `.cmd` shims like `typescript-language-server.cmd` resolve — std runs `.cmd`/`.bat` through `cmd.exe` automatically since Rust 1.77.2). `ServerProcess` spawns the server with piped stdio and a dedicated **reader thread** that frames and routes JSON-RPC traffic, mirroring the PTY reader-thread pattern from M3 — there is no async runtime inside the LSP module; request/response correlation uses `std::sync::mpsc`, and completion's fail-soft deadline uses `recv_timeout`. stderr is drained on its own thread (echoed sanitized in debug, discarded in release). Graceful `shutdown` (shutdown request → exit notification → wait → force-kill on timeout) and `Drop` (always kill + reap) guarantee no orphaned children, and a `shutting_down` flag stops an intentional teardown from being misreported as a crash.

`engine/src/lsp/client.rs` ties it together. `LspClient::connect` runs the `initialize`/`initialized` handshake (advertising full sync intent and `snippetSupport: false`), stores the server name, and flips status `Starting → Connected`; an unexpected process exit (detected by the reader thread hitting EOF) flips it to `Crashed` and clears pending requests. The language-agnostic `LspManager` owns the client map keyed by `(server_key, root)` — so TS and JS genuinely share one process per workspace — plus a `uri → client` index. It exposes `ensure_client`, `status_for_path`, `restart` (tears matching clients down; the UI re-opens documents to reconnect — never an automatic restart loop), and `shutdown_all`. The engine stays Tauri-free by taking two callbacks at construction (`on_diagnostics`, `on_status`) — the same `Box<dyn Fn>` push pattern the PTY uses.

`frontend/src/main.rs` registers `LspManager` as managed state in `setup()` (where the `AppHandle` is available), wiring the two callbacks to emit `lsp://status` and `lsp://diagnostics`. The `lsp_status` and `lsp_restart` commands landed here, and `RunEvent::ExitRequested` now calls `shutdown_all`.

### Wave 3 — Document sync, diagnostics, Tauri event surface

`LspClient` gained `did_open`/`did_change`/`did_close` plus per-document version tracking (monotonic, UI-generated) and a shared `ClientShared { versions, diagnostics }` so the reader-thread notification handler can **drop diagnostics for a closed document** (Requirement 5.4) without a back-reference into the client. `diagnostics.rs` gained `parse_publish_diagnostics` (normalizing range/severity/source/code — including numeric codes coerced to strings — with missing severity defaulting to `information`) and a `DiagnosticsStore` where an empty diagnostics array clears the entry and close removes it. `LspManager` got `open_document` (ensure → `didOpen` → register routing), `change_document`, and `close_document`. The three document commands were registered in `main.rs`, and `frontend/ui/src/lib/tauri/commands.ts` gained the typed `lsp_*` wrappers plus `onLspDiagnostics`/`onLspStatus` listeners and the full DTO/status type surface (`LspStatus` as a `state`-tagged union, `LspDiagnostic`, etc.).

### Wave 4 — CodeMirror diagnostics + status UI

`@codemirror/lint` was added. `editor/codemirror-setup.ts` now includes `lintGutter()` and exports `applyDiagnostics(view, diagnostics)`, which maps LSP UTF-16 line/character ranges to CodeMirror offsets (exact under full-document sync, clamped so a stale range can never throw) and dispatches `setDiagnostics` — squiggles are enabled on first use, gutter markers come from the gutter extension. A new `stores/lsp.ts` caches status-by-path and diagnostics-by-path, owns the per-path version counter, mirrors the engine's extension→language map, and registers the global event listeners (`initLsp`, called from `main.ts`). The tab store now fires `lspOpenPath` on open and `lspClosePath` on close (both fire-and-forget so opening a file is never blocked by server startup); `Editor.svelte` debounces `didChange` (250 ms, flushed on tab switch/teardown) and re-applies diagnostics to the mounted view whenever the store updates. `LspStatusIndicator.svelte` (added to the status bar) renders connected/starting/missing/crashed/disabled/unsupported/plain, shows error/warning counts when connected, and offers a **Restart** action on crash that also re-opens the active document.

### Wave 5 — Completion request + CodeMirror autocomplete

`completion.rs` gained `normalize_completion`, accepting both response shapes (`CompletionList { items }` and a bare `CompletionItem[]`), applying insert-text precedence (`textEdit.newText` → `insertText` → `label`), mapping `CompletionItemKind` to CodeMirror-friendly type strings, extracting plain-text documentation from string-or-`MarkupContent` (clamped), and — since we advertise `snippetSupport: false` — **falling back to the plain label for any item still flagged as a snippet** rather than inserting raw `${1:…}` placeholders. `LspClient::completion` requires a connected server and an open document and uses a 2 s fail-soft timeout; `LspManager::completion` maps every failure (missing/crashed/timeout/not-open) to an empty list so typing never freezes. The `lsp_completion` command was registered, `@codemirror/autocomplete` was added, and `codemirror-setup.ts` gained an async LSP completion source (via a `lspPath` facet carrying each document's path) that converts the cursor offset to an LSP position, flushes the latest text before asking, and returns no options for unsupported files / missing servers / timeouts. `completionKeymap` was added ahead of the default keymap so Enter/Tab accept an open popup and fall through to indent otherwise.

### Wave 6 — Settings UI, gated real-server smoke, final polish

`LspSettingsSection.svelte` (added to the settings page) exposes the minimal per-language controls — enable toggle, command, args — with the built-in defaults shown as placeholders (blank = default) and a note that uninstalled servers appear in the status bar rather than as errors; it stores no secrets. `EngineSettings` in `commands.ts` grew the typed `lsp` section. Gated real-server smoke tests cover `--version` for all four binaries and a full connect → (diagnostics) → completion → restart cycle for Rust, TypeScript, and Python — each **skips cleanly** when the binary is absent *or present-but-nonfunctional* (this session's environment has a rustup `rust-analyzer` proxy whose component isn't installed, which the tests detect and skip). Final polish confirmed no placeholder/stub comments, no M6.5 features, thin command wrappers, and `#[serde(default)]` on every new settings field.

## Why this approach

Hand-rolling the transport instead of pulling `lsp-types`/`tower-lsp` kept the engine's dependency graph — and the release binary — untouched, which is the whole point of the BYO decision: the editor stays lightweight and the user supplies the heavy toolchains. Reusing the M3 reader-thread + push-callback pattern (rather than introducing an async LSP stack) meant the new code reads like the terminal code already in the tree, and let the engine pieces be correct and unit-tested before any UI existed. Full-document sync trades a few extra bytes per keystroke (mitigated by debouncing) for provable correctness against three servers with very different behaviours — the right call for a foundation milestone, with the incremental path left open. Keying clients by `(server_key, root)` with TS/JS collapsed to one bucket matches how these servers actually run (one process per project), and routing changes/closes/completions through a `uri → client` index avoids re-deriving the root on every call. Making every degraded state — unsupported language, disabled, missing server, crash, timeout — a non-blocking status rather than an error is what keeps editing and saving fully usable when LSP isn't available.

## Impact

- GwenLand IDE now provides inline diagnostics (squiggles + gutter) and language-server autocomplete for Rust, TypeScript/JavaScript, and Python, using servers the user installs — with a compact status indicator and minimal settings, and graceful, non-blocking degradation when a server is missing, disabled, or crashes.
- **New (engine):** `engine/src/lsp/` — `mod.rs`, `language.rs`, `config.rs`, `root.rs`, `json_rpc.rs`, `process.rs`, `client.rs`, `diagnostics.rs`, `completion.rs`, `error.rs`. **Changed (engine):** `settings.rs` (+`lsp` section), `lib.rs` (+`pub mod lsp`). `engine/Cargo.toml` is **unchanged** — no new Rust dependencies.
- **New (frontend):** `frontend/ui/src/lib/stores/lsp.ts`, `components/LspStatusIndicator.svelte`, `components/LspSettingsSection.svelte`. **Changed (frontend):** `frontend/src/main.rs` (manager + 6 commands + events + exit shutdown), `tauri/commands.ts` (wrappers/listeners/types), `editor/codemirror-setup.ts` (lint + autocomplete), `components/Editor.svelte`, `StatusBar.svelte`, `SettingsPage.svelte`, `stores/tabs.ts`, `main.ts`. **Frontend deps added:** `@codemirror/lint`, `@codemirror/autocomplete` (each directly required by M6 scope).
- **Verification:** `cargo check --workspace` clean; `cargo test -p gwenland-engine` → **222 passing** (86 in the new `lsp` module: framing, ids, pending cleanup, root detection, URI round-trips, severity/diagnostics normalization, completion normalization + precedence, status transitions, version tracking, payload construction, crash + restart, and gated real-server smoke); `svelte-check` 0 errors / 0 warnings (141 files); `vite build` succeeds.
- **Binary size (release, `cargo tauri build`, x64, profile: `opt-level = "z"` + LTO + `strip`):** the application binary `GwenLand-IDE.exe` is **5,233,664 bytes (≈ 5.0 MiB)**; the bundled installers are **MSI 2,572,288 bytes (≈ 2.45 MiB)** and **NSIS `…-setup.exe` 1,929,576 bytes (≈ 1.84 MiB)**. M6 added **no new engine Rust dependencies**, so the binary stays within the lightweight-editor budget; the two new front-end CodeMirror packages (`lint`, `autocomplete`) live in the webview JS bundle (`index-*.js` ≈ 897 kB / 259 kB gzip), not the native binary. Full build ran clean in 2m 19s (exit 0).
- **Things to keep in mind:**
  - **Verified by tests, type-check, and build — not yet by a running GUI.** The `cargo tauri dev` diagnostics/autocomplete smoke listed in the wave checkpoints could not be observed in this (headless) session. The non-blocking guarantees hold by construction and unit tests, but a manual pass with a real, *functional* server installed (this box's `rust-analyzer` is a componentless rustup proxy) is the remaining hands-on confirmation.
  - **`lsp_completion` accepts but ignores `version`** — it's reserved for a future incremental-sync world; the autocomplete source flushes a `didChange` before requesting so the server already has the current text.
  - **Deferred to M6.5** (untouched here, by design): hover, go-to-definition, find-references, rename, code actions, workspace symbols, formatting, semantic tokens, inlay hints, signature help — plus incremental sync, multi-root workspaces, and remote/SSH LSP.
